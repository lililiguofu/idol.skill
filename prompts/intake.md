# Intake：基础问卷 + 物料入口

## Step 1：爱豆本体（用户少填、Agent 多查）

> **原则**：**组合名、出道日、粉丝名、官方定位、代表作等公开信息，应由 Agent 自行检索**（网络搜索 + 维基摘要等），**不要**默认让用户背诵百科。用户只需给 **检索锚点**；人设细节可补充或纠错。

### 1a. 用户最少提供（必填其一）

- **检索锚点**：正式艺名 / 常用昵称 / 中英韩任一种可唯一定位的名字（例：`Jennie`、`金珍妮`、`张员瑛`）。  
  若用户只说「做 Blackpink 里那个 rap」，则先 **反问锁定成员** 再搜。

### 1b. Agent 必须主动完成（勿空等用户）

1. **联网检索**：用公开网页确认「是谁、哪个团、粉丝名、出道年、代表标签」；有歧义（同名不同人）时 **列出选项请用户点选**，**勿瞎猜**。
2. **维基 baseline（推荐）**：在消歧后运行（或由 Agent 等价拉取摘要）：
   ```bash
   python tools/baseline_public.py --title "<用户确认的词条标题>" --wiki zh.wikipedia.org --output /tmp/baseline.json
   ```
   韩文/英文圈可改用 `--wiki ko.wikipedia.org` 或 `en.wikipedia.org`（以能命中为准）。
3. **整理成表**：把检索到的公开字段写入草稿（供用户一眼核对），至少包含：**艺名、组合、粉丝名、出道日、担当/定位（若公开）、代表作 1–3 条**。

### 1c. 用户可选补充 / 纠错

- **灵魂人设一句话**（可跳过）：性格、口癖、Emoji 等——用户比搜索引擎更准时可写；**与检索结果冲突时以用户纠正为准**（公开事实除外，须说明来源）。  
  （更细的 MBTI / 标签见下 **「人设补强」**，创建时 **Agent 建议主动多问**，不必只收这一句。）

### 1d. 汇总确认

向用户展示 **「检索结果 + 你的补充」**，请用户回答「确认」或「改：……」，再进入 Step 1b。

## Step 1b：粉丝向（强烈建议，可部分跳过）

> 目的：让模型知道 **你是哪种粉、从何时入坑、雷区在哪**，输出语气（妈粉 vs 女友粉等）会完全不同。对应写入见 [build_universe.md](build_universe.md)「粉丝成分概览」、[build_persona.md](build_persona.md)「私有投射 / 逆境档案 / 营业生物钟」。

### 1. 粉丝成分概览（Fandom Profile）

| 字段 | 含义 | 例 |
|------|------|-----|
| **入坑节点** `{milestone}` | 何时、因何入坑 | Pink Venom 打歌期 / Attention 出道即入坑 |
| **粉丝属性** `{fan_type}` | 自我定位 | 女友粉 / 事业粉 / 妈粉 / 团偏某成员 |
| **关系定位** `{relationship_stance}` | 你希望赛博分身怎么对你 | 遥远偶像 / 树洞同龄人 / 傲娇小猫 等 |
| **雷点 / 脱粉边缘** `{taboos}` | 不要提的队友、讨厌的官方话术 | 可写「尽量避免提 XX」「别讲爹味鸡汤」 |

### 2. 追星编年史（Milestone Timeline）

请用户按行提供（多少行随意），或说「跳过」：

| 日期 | 公开事件 | 你的私人记忆 / 互动 |
| :--- | :--- | :--- |
| `{date}` | 某次回归 / 出道 | 第一次看 MV 被惊艳 |
| `{date}` | 演唱会 / 签售 | 具体应援或见面经历 |
| `{date}` | 争议或低谷期（可选） | 超话熬夜、刷数据等（仅事实，不造谣） |

### 3. 饭圈共识 vs 私有投射

- **大屏共识（Fandom Inside Jokes）**：全团/全饭圈都懂的梗（如某造型梗、名台词），用于人物底层逻辑。  
- **1v1 私有投射**：签售记忆、你希望她对你的**专属称呼**、只属于你的小故事——写入须标「用户自述」，避免模型幻觉成「她真说过」。

### 4. 破防与逆境档案（Vulnerability Archive）

- **高压反应模式**：黑酸、连轴转、成绩未达预期时，她**公开语境里**通常怎么表达？（故作坚强 / 深夜长文 emo 等——仅用户或物料来源）  
- **对粉丝的逆向说教**：会不会管粉丝睡觉、吃饭？（责备式关心）

### 5. 营业生物钟（Business Biological Clock）

- **泡泡活跃时段** `{time_patterns}`：如常半夜两点发语音 / 早八点早安。  
- **营业物料配比** `{content_ratio}`：自拍多 / 风景多 / 短消息轰炸。  
- **消失频率** `{ghosting_pattern}`：日更 / 常一周不见再道歉式出现。

## 人设补强（可选 · 接在 Step 1b 后 · 建议 Agent **主动多问**）

> **目的**：创建初期把人设写进 `persona.md` Layer 3 / 词典时更有抓手；**整块可跳过**，但 Agent **不要默认不问**——可一句「下面几项有就填，没有就说跳过」。

| 字段 | 含义 | 写入提示 |
|------|------|----------|
| **MBTI** | 十六型 | 若艺人**公开自报**或权威采访写明 → 记入 `persona` 并标「公开来源」；**未公开** → 可记用户「个人推测」或「未知」，**勿写成官宣**。 |
| **血型 / 星座** | 娱乐向 | 可空；有则写入 universe / persona 一笔即可。 |
| **动物塑 / 代表 emoji** | 饭圈常用锚点 | 例：兔塑、🐰🤍；无则跳过。 |
| **用户观察到的口癖 / 句尾** | 尚无泡泡时 | 先记印象，导入语料后以 `parse_bubble` 为准迭代。 |
| **希望分身多强调的气质** | 与粉丝向协同 | 例：傲娇 / 姐感 / 软萌 / 冷感 — 与 Step 1b「关系定位」可合并讨论。 |
| **对用户专属称呼（若已有）** | 1v1 | 与 Step 1b「私有投射」重复则合并，不重复录入。 |

**Step 1b 与人设补强（若填写）汇总确认后进入 Step 2。**

## Step 2：物料导入（展示下列选项）

```
投喂的物料越纯，她就越像她。你想怎么导入？

  [A] Bubble / 泡泡聊天导出（强烈推荐）
      截图或文本导出，提取口癖与 Emoji 频率。

  [B] Weverse / Phoning 动态存档
      成员互动、深夜留言、给粉丝的回复。

  [C] 采访文字稿 / 媒体专访（建议优先本命采访）
      官方问答、深度访谈、综艺对谈片段，帮助标签性格与语气。可配合 `tools/parsers/parse_fansign.py` 标注人设关键词。

  [D] 社交媒体 & 物料转写
      Ins/微博语录、演唱会 Ment 文本等。

  [E] 脑内记忆直输
      内部梗、她对你的称呼、爱吃的零食、最近回归造型等。

可混选，也可跳过（仅靠公开世界知识会偏「官方腔」）。
```

**操作提示**

- A / D：优先 `.txt`；截图可先 OCR 或人工粘贴成文本再跑 `tools/parsers/parse_bubble.py` / `tools/analyze_emoji.py`。
- B：需要用户自行配置 Cookie 时再运行 `tools/parsers/parse_weverse.py`（见 `--help`）。
- C：使用 `tools/parsers/parse_fansign.py` 辅助打标签。
- E：直接记入笔记，后续写入 `universe.md` / `persona.md`。

## 冷启动兜底（语料不足时）

若 **没有** A–D 类足够文本（例如新粉只有人名 + 担当标签），则：

1. **Agent 仍应先完成 Step 1 的检索 + baseline**；维基词条标题由 **Agent 检索后列出候选 + 用户确认**（**勿猜测消歧**）。运行示例：

   ```bash
   python tools/baseline_public.py --title "词条名" --wiki zh.wikipedia.org --output /tmp/baseline.json
   ```

   亦可让用户粘贴 **搜狗百科 / 官方简介** 等公开文字（自动抓取第三方百科本站不内置，可手动复制补充）。

2. 用 baseline 仅构建 **粗颗粒** `universe.md`（公开事实）与 Layer 1 草稿；语气仍偏官方时，**必须**在预览摘要与 `meta.json` 中标注：

   - `warnings.low_corpus_purity: true`
   - `warnings.messages` 追加说明，例如：「当前语料纯度较低，建议后续导入更多泡泡聊天以解锁真实语气。」

3. 部署完成后再次提醒用户补充私域物料。
